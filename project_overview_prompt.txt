**THE DATA PIPELINE (The "Daily Check-in" Flow)**
Our flagship backend route is `POST /api/athlete/daily-checkin`. It follows a strict 3-phase enterprise pipeline:

1. **Ingestion:** The frontend sends a raw voice/text `transcript` and a self-reported `pain_level` (1-10) to the backend.
2. **Phase 1: Deterministic Red Flag Gateway (Middleware):** 
   - Before any AI is invoked, the system checks if `pain_level >= 8` or if the transcript contains critical medical keywords ("pop", "snap", "numbness", "cannot bear weight").
   - If triggered, the pipeline halts immediately and returns a hardcoded `CRITICAL` risk payload demanding immediate medical attention. This guarantees zero-latency for emergencies and prevents LLM hallucination on severe injuries.
3. **Phase 2: RAG Pipeline (Context Retrieval):**
   - **Symptom Parsing:** We use Gemini to extract structured JSON (`primary_location`, `pain_type`) from the transcript.
   - **Vector Embedding:** We generate an embedding for those symptoms using `text-embedding-004`.
   - **Firestore Vector Search:** We run a cosine similarity search (`findNearest`) against the `rehab_protocols` collection in Firestore to fetch the top 2 closest clinical protocols. (If unseeded, it falls back to a hardcoded `getMockProtocol` function).
4. **Phase 3: Master Inference:**
   - We fetch the athlete's biometric baseline (HRV, Sleep, RPE) from Firestore.
   - We concatenate the baseline, the parsed symptoms, and the retrieved RAG protocols into a massive prompt.
   - We invoke Gemini with a strict system instruction: *"ONLY recommend exercises explicitly stated in the retrieved clinical protocols."*
5. **Output:** A structured JSON response containing `risk_level`, `injury_probability_score`, `recovery_protocol`, and `training_adjustment`.

**FILE STRUCTURE & WHAT THEY DO**

**Backend (Node.js/Express):**
- `server/server.ts`: The main Express server entry point. It initializes Firebase Admin and mounts the API routers (`/api/analysis` and `/api/athlete`).
- `server/routes/athlete.ts`: **(The Core Engine)** Contains the newly refactored `POST /daily-checkin` endpoint. This file houses the Red Flag Gateway (`checkRedFlags`), the RAG context fetcher (`fetchVectorContext`), and the Master Inference wrapper (`callGeminiInference`).
- `server/seedFirestore.ts`: A standalone data engineering script. It generates vector embeddings for our mock clinical protocols and seeds them into the Firestore database.
- `server/routes/analysis.ts`: The legacy AI router that uses the Anthropic Claude SDK for endpoints like `/symptoms`, `/daily`, `/team`, and `/image` (Claude Vision for MRI/Scan analysis).
- `server/lib/prompts.ts`: Contains the legacy prompt building logic for the Anthropic endpoints.

**Frontend (React/Vite):**
- `src/App.tsx` & `src/main.tsx`: The React entry points and routing structure.
- `src/components/`: Modular React components for the landing page and application interface.
- `src/services/`: Client-side logic for connecting to Firebase (`firebase.ts`), local mock data (`db.ts`), and the backend REST API (`api.ts`).
- `stitch/`: A large directory containing highly-polished static HTML/CSS UI components (e.g., `athlete_dashboard_aesthetic`, `recovery_insights`) generated via the Google Stitch MCP for UI consistency.
